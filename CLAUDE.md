## 1. 프로젝트 한 줄 정의
**ASTRA** — 장착 무기를 통째로 갈아끼우며 수많은 무기를 시험하는 **종스크롤 파괴 쾌감형 슈팅**. (Unity, C#)
- **장르:** 종스크롤(위→아래) 정통파, 파괴 쾌감형
- **레퍼런스:** 라이덴, 스트라이커즈 1945 (화력 중심) + B-WINGS (교체 모듈형 무기)
- **차별점:** 파워업 누적이 아닌 **무기 통째 교체**. 무기·적·보스를 기획자가 코드 없이 찍어내는 **데이터 주도 + 커스텀 에디터** 파이프라인.

## 2. 확정 핵심 스펙

### 게임 정체성 · 스크롤 [확정]
- 종스크롤(위→아래). 재미의 중심 = 쏘는 재미(파괴 쾌감). — GDD §0.1, §0.2

### 지형 [확정]
- **하이브리드** — 주로 개활지 + 일부 지형 구간. 책 **다중 스크롤**(Ch.7 §7.4)로 구현: 깊은 층=우주(충돌X) / 얕은 층=지형(타일맵, 충돌O). — GDD §6.0

### 무기 시스템 [확정] — GDD §3
- **교체 모듈형**: 슬롯 1개, 무기 줍는 즉시 **즉시 교체**.
- **픽업 2종**: 무기 아이템(W=교체) / 파워업 아이템(P=강화).
- **무기-출력(파워 레벨)**: **Player(기체)의 속성**, 최대 4단계. 무기 교체해도 **유지**, **사망 시 리셋**.
- **기본 무기**: 정면 연사(Vulcan), 항상 보유.
- 강화 곡선 3축(레벨별): 동시 발사 수 · 위력 · 연사 속도. (탄 범위/관통은 무기 고유, 레벨 비례 아님)

### 기술 사양 [잠정 — 구현 중 확정]
- 프레임: **60fps 고정** [잠정 권장]
- 좌표계: Unity 2D 월드 좌표. **Y+ = 위(전진 방향)**. 1유닛=?px [잠정]
- 충돌 판정: **원-원(거리 비교)** 기본 [잠정]. 사각형×복합 정도로 제한(복합×복합 회피, §5.19)
- 해상도/플레이 영역: 종형 [잠정]

### 미정(구현 중 채움)
- 플레이어 이동감·라이프·옵션(§2), 특수 공격(§4), 스코어링·난이도·아이템·플로우(§7), 사운드·연출(§8)

## 3. 아키텍처 규칙 [확정]

### 3.1 데이터 주도 설계 (최우선 원칙)
- **모든 콘텐츠(무기·적·편대·보스·스테이지)는 ScriptableObject "데이터 카드"로 정의.** 코드에 하드코딩 금지.
- 콘텐츠 추가 = **카드 추가**(기존 동작 재사용 시 코드 무수정, 신규 동작만 작은 스크립트 1개 추가).

### 3.2 네임스페이스
- `ShootingGame.[기능명]` 규칙. 예: `ShootingGame.Bullet`, `ShootingGame.Player`, `ShootingGame.Weapon`, `ShootingGame.Enemy`, `ShootingGame.Boss`, `ShootingGame.Stage`, `ShootingGame.Item`, `ShootingGame.Core`.
- 에디터 코드: `ShootingGame.EditorTools` (Editor 전용 어셈블리).

### 3.3 ScriptableObject 카드 (데이터 모델)
| 카드 | 역할 | GDD |
|------|------|-----|
| `WeaponData` | 무기 특성 + 무기-출력 Lv.1~4 강화 곡선 | §3.2 |
| `EnemyData` | 개체 적 특성 + 이동 방식(`moveType`) | §5.1 |
| `FormationData` | 스폰 시 편대(무리) 구성 | §5.1 |
| `BossData` | 파츠 + 페이즈(FSM) + 기믹 | §5.2 |
| `StageData` | 배경(다중 스크롤) + 구간 + 스폰 타임라인 | §6.1 |

### 3.4 커스텀 에디터 (4종, EditorWindow)
- 무기 에디터 / 적기·편대 에디터 / 보스 에디터 / 레벨 에디터.
- 각 에디터는 위 카드를 기획자가 코드 없이 저작. `Editor/` 폴더(에디터 전용 어셈블리)에 구현.

### 3.5 FSM (상태 머신)
- 상태 기반 행동은 FSM으로. 적용처: 보스 페이즈(§5.2), 적 등장(`READY→APPEAR→ACT→DISAPPEAR`, 책 §6.4), 플레이어 상태 등.

### 3.6 오브젝트 풀링 (필수)
- 탄환·적·이펙트 등 대량 생성/소멸 객체는 **반드시 풀링**. `Instantiate/Destroy` 남발 금지 → GC 스파이크 방지, 60fps 유지.

## 4. 폴더 / 파일 구조 컨벤션

프로젝트 루트의 문서 배치는 §0 참조. 아래는 `Assets/` 내부 구조.

```
Assets/
  ScriptableObjects/         # 데이터 카드(.asset)
    Weapons/  Enemies/  Formations/  Bosses/  Stages/  Items/
  Scripts/
    Core/      # 게임 루프, 오브젝트 풀, 입력, 매니저
    Player/    # 기체, 무기-출력 보유, 피탄
    Bullet/    # 탄환(자기탄/적탄), 풀링
    Weapon/    # 무기 동작(WeaponData 해석), 픽업/교체
    Enemy/     # 개체 적, 이동(moveType), 편대
    Boss/      # 파츠, 페이즈 FSM, 기믹
    Stage/     # 배경 다중 스크롤, 구간, 스폰 타임라인
    Item/      # W/P/점수 드롭
    Editor/    # 4대 커스텀 에디터(EditorWindow)
  Art/  Audio/  Scenes/  Tilemaps/
```

### 네이밍 규칙
- 타입/메서드: `PascalCase` · 필드: `camelCase` · 상수: `UPPER_SNAKE`.
- ScriptableObject 클래스: `...Data` 접미사. 카드 에셋 파일명: `W02_Laser.asset`, `E01_Straight.asset` 등 식별 가능하게.
- MonoBehaviour(런타임 동작) ↔ ScriptableObject(데이터) 역할을 명확히 분리.

### 의존 패키지 [잠정]
- Unity 2D 기본, 2D Tilemap(지형 구간), TextMeshPro(UI). New Input System 사용 여부 구현 중 결정.

## 6. 코드 변환 규칙 (책 C/C++ + DirectX → Unity C#)
- DirectX 좌표계 → **Unity 월드 좌표**로 변환(그대로 옮기지 말 것). 화면 좌표는 스크롤 위치 기준 상대 좌표(책 §7).
- C 배열 → `List<T>` 또는 Unity 네이티브 컨테이너.
- 전역 변수 → ScriptableObject 또는 MonoBehaviour 멤버.
- 게임 루프 직접 구현 → `Update()` / `FixedUpdate()`.
- 직접 드로잉 → `SpriteRenderer` / `Transform`.
- 삼각함수·벡터 → `Mathf`, `Vector2/3`.

## 7. 하지 말 것 / 주의
- ❌ 무기·적·보스·스테이지 수치를 **코드에 하드코딩** → 반드시 데이터 카드(SO)로.
- ❌ 새 콘텐츠 추가 시 기존 카드/코드 **대규모 수정** → 추가(add)식으로.
- ❌ 탄환·적 등 대량 객체 `Instantiate/Destroy` 남발 → 풀링.
- ❌ **무기-출력을 무기 측에 저장** → 기체(Player) 속성이 정답(교체해도 유지).
- ❌ 복합×복합 충돌 판정 → 비용 과다(§5.19). 사각형×복합으로 제한.
- ⚠️ 보스는 "부품의 집합"으로 설계 → 부위 파괴·분리·합체·변신이 자연스러워짐(책 §6.10~13).
- ⚠️ 보스 AI는 "영리함"보다 **규칙적이고 학습 가능한 패턴**(책 §6.11).
